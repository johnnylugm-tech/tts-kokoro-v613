## 7. 兩次 A/B 審查流程

### 第一次審查（TEST_PLAN，執行前）

```
Agent A (qa) 撰寫 TEST_PLAN.md
         ↓
Agent B (reviewer) 審查 TEST_PLAN.md
         ↓
     ❌ REJECT → Agent A (qa) 修訂 → 重新審查
         ↓
     ✅ APPROVE → 才能執行測試
```

### 第二次審查（TEST_RESULTS，執行後）

```
Agent A (qa) 執行 pytest + 記錄 TEST_RESULTS.md
         ↓
Agent B (reviewer) 審查 TEST_RESULTS.md
         ↓
     ❌ REJECT → Agent A (qa) 修復 → 重新審查
         ↓
     ✅ APPROVE → Quality Gate → sign-off
```

## 8. TEST_PLAN.md 規格

### 必須包含

1. **測試目標與範圍** — 驗證所有 FR 符合 SRS
2. **測試策略與環境** — pytest + mock + ffmpeg
3. **TC 清單** — 每個 FR ≥1 個 TC，P0 需求（FR-01~03）有正向+邊界+負面三類
4. **風險評估** — Mock 策略、外部依賴
5. **FR↔TC 映射表** — 每個 FR 對應哪些 TC

### 設計原則

- **從 SRS 推導 TC**，不是從代碼推導
- **不看 Phase 3 代碼**來設計測試案例
- TC 命名：`TC-FR{XX}-{TYPE}-{NN}`（如 `TC-FR01-POS-01`）

---

## 9. TEST_RESULTS.md 規格

### 必須包含

1. **執行摘要** — 日期、總數、通過率、覆蓋率
2. **詳細結果** — 每個 FR 的測試數和狀態
3. **失敗案例分析** — 根本原因（具體到模組+函數+行號）
4. **覆蓋率報告** — pytest-cov 自動生成數據
5. **實際 pytest 輸出** — 不是摘要數字

### ⚠️ 必須有實際輸出

- pytest stdout/stderr（不是摘要數字）
- pytest-cov coverage 報告（不是估算）

## 11. sessions_spawn.log 格式（HR-10）⚠️ 重要

**⚠️ 即時寫入，不是完成後補**

每次 `sessions_spawn()` 呼叫時，**在派遣前**寫入：

```json
{"timestamp":"ISO8601","role":"qa","task":"任務名稱","session_id":"agent:...","fr":"PHASE4","confidence":0,"verdict":"N/A"}
```

派遣完成後更新 verdict：

```json
{"timestamp":"ISO8601","role":"qa","task":"任務名稱","session_id":"agent:...","fr":"PHASE4","confidence":1-10,"verdict":"PASS|APPROVE|REJECT"}
```

### ⚠️ 角色名稱（Auditor 識別）

| 正確 | 錯誤 |
|------|------|
| `qa` | `developer` |
| `qa` | `manager` |
| `reviewer` | ✅ |

**嚴禁**：將 `manager` 或 `developer` 寫入 sessions_spawn.log
